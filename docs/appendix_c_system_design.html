รูปที่ ค-4: Sequence Diagram แสดงกระบวนการ Form-Based Authentication และการใช้ Session Cookies
คำอธิบาย
  • Auto-detect form fields: ระบบค้นหา username/password field ด้วยลำดับความสำคัญ 4 ระดับ ไม่จำเป็นต้องกำหนด field name เอง
  • CSRF Token: ระบบ parse <input type="hidden"> ทุกตัวในฟอร์ม และส่งไปพร้อม POST เพื่อ bypass CSRF protection
  • Shared CookieJar: httpclient.SetSessionJar() ทำให้ plugin ทุกตัวใช้ session เดียวกัน ผ่าน HTTP client ที่แชร์ร่วมกัน
  • Cleanup: defer SetSessionJar(nil) ล้าง session เมื่อ Scan เสร็จ เพื่อไม่ให้ session หลุดไปยัง Scan ครั้งถัดไป

SD-03: การทดสอบ Brute Force Credential Attack

sequenceDiagram autonumber participant ENG as Scanner Engine participant BF as runBruteForce() participant TGT as Target Login Endpoint participant RPT as Findings Collector ENG->>BF: runBruteForce(ctx, bruteForceConfig) BF->>TGT: GET {login_url} [ดึง form สำหรับ auto-detect fields] TGT-->>BF: HTML form loop สำหรับแต่ละ (username × password) pair

รูปที่ ค-5: Sequence Diagram แสดงกระบวนการ Brute Force Testing ด้วย 3-Signal Detection
คำอธิบาย 3-Signal Detection
  • Signal 1 — Status Code: หาก login สำเร็จ server มักตอบด้วย 302 Found (redirect) แทน 200 หรือ login page ซ้ำ
  • Signal 2 — Body Change: เปรียบเทียบ response body size/content กับ baseline (failed login) หากต่างกันมากกว่า threshold → probable success
  • Signal 3 — Redirect Target: ตรวจว่า Location header ชี้ไปยัง URL ที่ต่างจาก login page (เช่น /dashboard)
  • Safety caps: MaxAttempts สูงสุด 500, DelayMs ต่ำสุด 100ms เพื่อป้องกัน accidental DoS

คำอธิบายเพิ่มเติม
  • Phase ของ Progress: ระบบรายงาน 4 phases — idle (0%), crawling (5%), scanning (10%–95%), reporting (98%)
  • Log Streaming: GET /logs?after=N ส่งคืนเฉพาะ log lines ใหม่ที่เพิ่มขึ้นหลัง index N ทำให้ไม่ต้องส่ง log ซ้ำ
  • Global Cancel: engine.SetGlobalCancel() เก็บ cancel function ไว้ใน global variable ที่ป้องกันด้วย mutex ทำให้ POST /cancel สามารถหยุด Scan ได้จากภายนอก
  • TimedMode: หาก Scan mode = Timed จะใช้ context.WithTimeout() แทนการ cancel ด้วยมือ โดย timeout สูงสุดคือ 3600 วินาที

อธิบายการทำงาน Step by Step
  1. ขั้นที่ 1 — รับและตรวจสอบ Request: ระบบรับ ScanRequest จาก GUI แล้วตรวจสอบและปรับค่าให้อยู่ในขอบเขตที่ปลอดภัย ได้แก่ MaxPages (1–200), MaxDepth (1–4) และ RequestDelayMs (0–500 ms)
  2. ขั้นที่ 2 — เลือก Scan Mode: ระบบแยกเส้นทางการทำงานตามโหมดที่เลือก โดย Normal Mode จำกัดหน้าและความลึก, Timed Mode ใช้ time limit สูงสุด 3,600 วินาที และ Full Scan Mode ไม่จำกัดทั้งหน้าและความลึก
  3. ขั้นที่ 3 — Authentication (ถ้าเปิดใช้งาน): ระบบเรียก performLogin() เพื่อ login ไปยังเว็บเป้าหมายและเก็บ session cookies ไว้ใน CookieJar ซึ่ง HTTP client ทุกตัวในระบบจะนำไปใช้ร่วมกันโดยอัตโนมัติ
  4. ขั้นที่ 4 — Brute Force (ถ้าเปิดใช้งาน): ระบบเรียก runBruteForce() เพื่อทดสอบรหัสผ่านจากรายการที่กำหนด โดยใช้ 3-Signal Detection และบันทึกผลเป็น Critical findings ก่อนเริ่ม crawl
  5. ขั้นที่ 5 — Crawling: Crawler ทำ BFS โดยเริ่มจาก URL เป้าหมายและขยายไปยัง URL อื่น ๆ ที่อยู่บน hostname เดียวกัน ผลลัพธ์คือ []URLInfo ที่มี URL และ depth ของแต่ละหน้า
  6. ขั้นที่ 6 — Parallel Scanning: ระบบสร้าง goroutine pool (สูงสุด 10 ตัวพร้อมกัน) โดยแต่ละ goroutine รับ URL หนึ่งหน้าแล้วส่งให้ plugin ทุกตัวที่เปิดใช้งานทำการตรวจสอบตามลำดับ ผลลัพธ์จากทุก goroutine ถูกรวมเข้าใน shared slice ที่ป้องกันด้วย mutex
  7. ขั้นที่ 7 — Deduplicate: ระบบกรอง findings ที่ซ้ำกันออกโดยใช้ key ที่ประกอบด้วย Type|TargetURL|Evidence เพื่อให้รายงานไม่มีช่องโหว่ซ้ำซ้อน
  8. ขั้นที่ 8 — คำนวณ Risk Score: ระบบเรียก calcRiskScores() เพื่อคำนวณคะแนนความเสี่ยงแบบไดนามิกให้กับ findings แต่ละประเภท โดยใช้สูตร OWASP-Inspired ที่อ้างอิง incidence rate, exploit score และ impact score
  9. ขั้นที่ 9 — เรียงลำดับ: ระบบเรียง findings ตาม RiskScore จากมากไปน้อย โดยใช้ CVSSScore เป็นตัวแบ่งเมื่อคะแนนเท่ากัน ทำให้ช่องโหว่ที่รุนแรงที่สุดปรากฏก่อนเสมอ
  10. ขั้นที่ 10 — สร้างรายงาน: ระบบเรียก GenerateArtifacts() เพื่อสร้างไฟล์รายงานในรูปแบบ JSON, HTML และ PDF พร้อม Executive Summary, Methodology, Findings Table และ Conclusions บันทึกไว้ใน reports/{scan_id}/
  11. ขั้นที่ 11 — ส่งผลกลับ: ระบบส่ง ScanResponse ที่ประกอบด้วย scan_id, findings, สถิติ (Critical/High/Medium/Low) และ path ของไฟล์รายงานกลับไปยัง GUI ผ่าน HTTP 200

รูปที่ ค-8: Flowchart แสดงอัลกอริทึม BFS (Breadth-First Search) ของ Web Crawler รองรับ SPA Hash Routes
อธิบายการทำงาน Step by Step
  1. ขั้นที่ 1 — เริ่มต้น: กำหนดค่าเริ่มต้นโดยใส่ StartURL ลงใน queue ที่ depth=0 พร้อมสร้าง map visited สำหรับป้องกัน URL ซ้ำ และ slice results สำหรับเก็บผลลัพธ์
  2. ขั้นที่ 2 — ตรวจสอบเงื่อนไขหยุด: ก่อนประมวลผล URL แต่ละตัว ระบบตรวจสอบว่า queue ว่างหรือจำนวน results ถึง MaxPages แล้วหรือไม่ ถ้าใช่ให้หยุดและคืนค่า results ทันที
  3. ขั้นที่ 3 — นำ URL ออกจาก Queue: Dequeue URL ตัวแรกจาก queue แล้วทำ URL normalization โดยตัด fragment anchor (#section) และ session/token query parameters ออก เพื่อให้ได้ key สำหรับตรวจสอบ URL ซ้ำ
  4. ขั้นที่ 4 — ตรวจสอบ URL ซ้ำ: หาก URL นี้เคยถูก visit แล้ว ให้ข้ามไปขั้นที่ 2 โดยไม่ประมวลผล มิฉะนั้น mark ว่า visited และเพิ่มลง results
  5. ขั้นที่ 5 — ตรวจสอบ Depth: หาก depth ของ URL นี้ถึง MaxDepth แล้ว ให้หยุดขยาย link จาก URL นี้ (เพิ่ม URL ใน results แล้วแต่ไม่ enqueue link ต่อ) กลับไปขั้นที่ 2
  6. ขั้นที่ 6 — ดึง Links: เรียก fetchLinks() เพื่อส่ง HTTP GET ไปยัง URL นั้น แล้ว parse HTML เพื่อดึงทุก href= ออกมา รวมทั้ง SPA hash routes (href="#/route") โดยข้าม javascript: และ mailto:
  7. ขั้นที่ 7 — กรองแต่ละ Link: สำหรับแต่ละ link ที่พบ ระบบแปลงเป็น absolute URL จากนั้นกรอง 3 เงื่อนไข ได้แก่ (1) ต้องอยู่บน hostname เดียวกับ StartURL, (2) ยังไม่ถูก visit และ (3) ไม่ใช่ SPA hash route
  8. ขั้นที่ 8 — จัดการ SPA Hash Route: หาก link เป็น SPA hash route (fragment ขึ้นต้นด้วย "/") ให้เพิ่ม URL นั้นลง results โดยตรงโดยไม่ต้อง fetch ใหม่ เนื่องจาก server คืน HTML เดิมสำหรับทุก hash route
  9. ขั้นที่ 9 — Enqueue: หาก link ผ่านทุกเงื่อนไข ให้ enqueue โดยเพิ่ม depth+1 เพื่อรอประมวลผลในรอบถัดไป แล้ววนกลับไปขั้นที่ 2
คำอธิบายพิเศษ
  • SPA Hash Route: เว็บ framework อย่าง Angular, React, Vue ที่ใช้ HashRouter จะมี URL รูปแบบ /#/dashboard ระบบตรวจจับและเพิ่ม URL เหล่านี้ลง results โดยตรงโดยไม่ต้อง fetch ใหม่ เพราะ server คืน HTML เดิมเสมอ
  • URL Normalization: ตัด fragment (#section), session token query params เพื่อไม่นับ URL เดียวกันหลายครั้ง แต่คง SPA fragment (#/route) ไว้
  • Same-host filter: เปรียบเทียบ hostname (case-insensitive) ป้องกันการ crawl ออกนอก domain เป้าหมาย

รูปที่ ค-9: Flowchart แสดงการทำงานของ XSS Scanner ด้วย Nonce Probe และ CSP Detection
อธิบายการทำงาน Step by Step
  1. ขั้นที่ 1 — รับ URLInfo และ Parse: XSSScanner รับ URL จาก Engine แล้ว parse เพื่อดึง query parameters ทั้งหมด หาก URL ไม่มี query parameters ให้หยุดและคืนค่า nil ทันที เพราะ reflected XSS ต้องการ input vector
  2. ขั้นที่ 2 — สร้าง Nonce Probe: ระบบสร้างสตรีงสุ่ม 8 ตัวอักษร (nonce) เช่น a1b2c3d4 ที่ไม่ใช่ XSS payload แต่ใช้เพื่อตรวจสอบว่าค่าที่ส่งไปถูก reflect กลับมาในหน้าเว็บหรือไม่โดยไม่ทริกเกอร์ WAF
  3. ขั้นที่ 3 — ทดสอบแต่ละ Parameter: สำหรับแต่ละ query parameter ระบบแทนที่ค่าด้วย nonce probe แล้วส่ง GET request ไปยัง URL ที่แก้ไขแล้ว
  4. ขั้นที่ 4 — ตรวจสอบ Reflection: ระบบตรวจสอบว่า nonce ปรากฏใน response body หรือไม่ หากไม่พบ แสดงว่า server ไม่ได้ reflect input กลับมา จึงข้ามไปทดสอบ parameter ถัดไป
  5. ขั้นที่ 5 — ตรวจสอบ HTML Context: หาก nonce ถูก reflect กลับมา ระบบตรวจสอบว่าค่านั้นอยู่ใน HTML context ที่อันตรายหรือไม่ เช่น ภายใน script tag, attribute value หรือ event handler ถ้าถูก encode/escape แล้วถือว่าปลอดภัยและข้ามไป
  6. ขั้นที่ 6 — ตรวจสอบ CSP Header: ระบบตรวจสอบว่า response มี Content-Security-Policy header หรือไม่ ถ้ามี CSP จะสร้าง Finding ระดับ Medium เนื่องจาก CSP ช่วยลดความเสี่ยง ถ้าไม่มี CSP จะสร้าง Finding ระดับ High
  7. ขั้นที่ 7 — FullScan Mode (ถ้าเปิด): หาก Scan Mode เป็น Full Scan ระบบจะทดสอบ XSS payload จริงเพิ่มเติม เช่น script alert payload, img onerror payload และ svg onload payload เพื่อยืนยันว่าช่องโหว่สามารถ exploit ได้จริง
  8. ขั้นที่ 8 — ตรวจสอบ Payload Reflection: หาก payload ถูก reflect กลับมาโดยไม่ถูก sanitize ระบบอัปเดต evidence ใน Finding ด้วย payload จริง เพื่อให้รายงานมีหลักฐานชัดเจน
  9. ขั้นที่ 9 — คืนค่า Findings: หลังทดสอบครบทุก parameter แล้ว คืนค่า findings ทั้งหมดที่พบกลับไปยัง Engine

รูปที่ ค-10: Flowchart แสดงการทำงานของ SQLi Scanner แบบ 3 Phase — Error, Boolean, และ Time-Based
อธิบายการทำงาน Step by Step
  1. ขั้นที่ 1 — ตรวจสอบ Input Vector: SQLiScanner รับ URL จาก Engine แล้วตรวจสอบว่ามี query parameters หรือ form fields ที่สามารถใช้เป็นช่องทาง injection ได้หรือไม่ ถ้าไม่มีให้หยุดทันที
  2. ขั้นที่ 2 — Phase 1 (Error-Based): ระบบ inject single quote (') เข้าไปในแต่ละ parameter แล้วส่ง request ไปยังเซิร์ฟเวอร์ จากนั้นตรวจสอบ response body ว่ามี SQL error message ปรากฏหรือไม่ เช่น ORA-, MySQL Error, Syntax Error หรือ SQLSTATE ซึ่งครอบคลุม MySQL, PostgreSQL, Oracle, MSSQL และ SQLite
  3. ขั้นที่ 3 — ผล Phase 1: หากพบ SQL error message ระบบสร้าง Critical finding ประเภท Error-Based พร้อม evidence ที่เป็น error message จริงจากเซิร์ฟเวอร์ จากนั้นดำเนินการต่อด้วย Phase 2 เพื่อหาช่องโหว่เพิ่มเติม
  4. ขั้นที่ 4 — Phase 2 (Boolean-Based): ระบบ inject TRUE condition (เช่น 1=1) แล้วบันทึก response A จากนั้น inject FALSE condition (เช่น 1=2) แล้วบันทึก response B เปรียบเทียบความแตกต่างของทั้งสอง response
  5. ขั้นที่ 5 — ผล Phase 2: หาก response A และ B แตกต่างกันอย่างมีนัยสำคัญ (เช่น ขนาด content ต่างกันเกิน threshold) แสดงว่า SQL query ถูกควบคุมได้ ระบบสร้าง Critical finding ประเภท Boolean-Based
  6. ขั้นที่ 6 — Phase 3 ตรวจสอบ Mode: Phase 3 จะทำงานเฉพาะเมื่อ Scan Mode เป็น Full Scan เท่านั้น เนื่องจากการ inject time-delay อาจทำให้การ scan ช้ามากหากทำกับทุก URL ใน Normal/Timed mode
  7. ขั้นที่ 7 — Phase 3 (Time-Based): ระบบ inject time-delay payload เช่น SLEEP(5) สำหรับ MySQL หรือ WAITFOR DELAY สำหรับ MSSQL แล้วส่ง request พร้อมวัดเวลาที่ใช้ในการได้รับ response
  8. ขั้นที่ 8 — ผล Phase 3: หาก response ใช้เวลามากกว่า 4.5 วินาที (หรือค่า threshold ที่กำหนด) แสดงว่าเซิร์ฟเวอร์รัน SQL delay function สำเร็จ ระบบสร้าง Critical finding ประเภท Time-Based ซึ่งตรวจจับได้แม้ response body ไม่มีความแตกต่าง
  9. ขั้นที่ 9 — คืนค่า Findings: รวม findings จากทุก phase ที่พบแล้วคืนค่ากลับไปยัง Engine
รายละเอียดแต่ละ Phase
  • Phase 1 (Error-based): เร็วที่สุด ตรวจจับจาก SQL error message ที่ server ส่งกลับมา ครอบคลุม MySQL, PostgreSQL, Oracle, MSSQL, SQLite error patterns
  • Phase 2 (Boolean-based): ตรวจจับโดยเปรียบเทียบ response ของ true/false condition หาก response ต่างกันแสดงว่า SQL query ถูก inject สำเร็จ
  • Phase 3 (Time-based): ทำเฉพาะ FullScan mode เท่านั้น ใช้ SLEEP() หรือ WAITFOR DELAY เพื่อทำให้ server ตอบช้าโดยตั้งใจ ตรวจจับได้แม้ response body ไม่เปลี่ยน

รูปที่ ค-11: Flowchart แสดงอัลกอริทึมการคำนวณ Dynamic Risk Score ด้วย OWASP-Inspired Formula
อธิบายการทำงาน Step by Step
  1. ขั้นที่ 1 — ตรวจสอบข้อมูลเบื้องต้น: ฟังก์ชัน calcRiskScores() รับ findings ทั้งหมดที่ผ่านการ deduplication แล้ว พร้อม totalURLs ที่ crawl ได้ หาก totalURLs เป็น 0 หรือไม่มี findings ให้คืนค่าเดิมโดยไม่คำนวณ
  2. ขั้นที่ 2 — จัดกลุ่ม Findings ตาม Title: ระบบวน loop ผ่าน findings ทั้งหมดเพื่อสร้าง map โดยใช้ Title เป็น key เก็บข้อมูล 2 อย่างสำหรับแต่ละ title คือ (1) จำนวนครั้งทั้งหมดที่พบ (occurrences) และ (2) set ของ URL ที่ถูกผลกระทบ (urlsAffected)
  3. ขั้นที่ 3 — คำนวณ IncidenceRate%: สำหรับแต่ละ title ระบบคำนวณอัตราการพบช่องโหว่โดยหาร จำนวน URL ที่ถูกผลกระทบ ด้วย totalURLs แล้วคูณ 100 เพื่อแปลงเป็นเปอร์เซ็นต์ เช่น พบใน 3 จาก 10 URL = 30%
  4. ขั้นที่ 4 — คำนวณ OccurrenceRatio: ระบบหาร occurrences ทั้งหมดด้วย totalURLs เพื่อได้อัตราส่วนการปรากฏซ้ำ ซึ่งแยกจาก IncidenceRate เพื่อจับกรณีที่ช่องโหว่เดียวปรากฏหลายครั้งในหน้าเดียวกัน
  5. ขั้นที่ 5 — ดึง ExploitScore และ ImpactScore: ระบบดึง ExploitScore และ ImpactScore จาก finding ตัวแรกของแต่ละ title เนื่องจาก findings ที่มี title เดียวกันจะมีค่าเหมือนกันเสมอ (กำหนดตาม severity tier)
  6. ขั้นที่ 6 — คำนวณ RiskScore: ระบบคำนวณ RiskScore ด้วยสูตร
    (IncidenceRate% × 0.30) + (ExploitScore × 10.0) + (ImpactScore × 20.0) + (OccurrenceRatio × 10.0)
    โดย ImpactScore ได้รับน้ำหนักสูงสุด (×20) สะท้อน OWASP methodology ที่เน้นผลกระทบมากกว่าความง่ายในการโจมตี
  7. ขั้นที่ 7 — ปัดทศนิยม: ปัดค่า RiskScore ให้เหลือ 2 ตำแหน่งทศนิยมด้วย banker's rounding เพื่อความสม่ำเสมอในการแสดงผล
  8. ขั้นที่ 8 — กำหนดคะแนนกลับ: ระบบวน loop ผ่าน findings ทั้งหมดอีกครั้งและกำหนด RiskScore จาก scoreByTitle[f.Title] ให้กับ findings ทุกตัวที่มี title เดียวกัน ทำให้ findings กลุ่มเดียวกันได้คะแนนเท่ากันเสมอ
  9. ขั้นที่ 9 — คืนค่า: คืน findings ทั้งหมดพร้อม RiskScore ที่คำนวณแล้ว กลับไปยัง Engine เพื่อใช้ในการเรียงลำดับและสร้างรายงานต่อไป
เหตุผลของ weight ในสูตร
  • ImpactScore × 20 — ให้น้ำหนักมากที่สุด สะท้อน OWASP methodology ที่ถือว่า "ผลกระทบ" สำคัญกว่า "ความง่ายในการโจมตี"
  • ExploitScore × 10 — น้ำหนักปานกลาง แทน exploitability sub-score ของ CVSSv3
  • IncidenceRate% × 0.30 — น้ำหนักน้อยที่สุด แทน "กี่เปอร์เซ็นต์ของ URL ที่ได้รับผลกระทบ" ใน scan นี้ (ปรับจาก OWASP population-level metric)
  • OccurrenceRatio × 10 — คำนวณจากจำนวนครั้งที่พบหาร totalURLs แยกจาก IncidenceRate เพื่อจับ findings ที่ปรากฏซ้ำหลายครั้งในหน้าเดียวกัน
  • Scale สูงสุด ≈ 340 — Critical ที่ 100% incidence ได้คะแนน 326, High=275, Medium=220, Low=150 ทำให้ severity เป็นตัวกำหนดหลักเสมอ
